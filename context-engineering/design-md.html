DESIGN.md 완전 가이드

코드가 "무엇"을 하는지는 읽으면 안다. 하지만 "왜" 그렇게 만들었는지는
디자인 문서에 적어두지 않으면 사라진다.

DESIGN.md란?

DESIGN.md는 프로젝트의 아키텍처, 설계 결정, 그 근거를 기록하는 마크다운 문서다. Google이 오래전부터 실천해온 "코딩 전에 디자인 문서를 먼저 쓴다"는 문화에서 유래하며, AI 코딩 도구의 등장으로 그 가치가 더 커졌다.

"디자인 문서의 가장 큰 가치는 코드를 작성하기 전에 생각을 강제한다는 점이다. 두 번째 가치는 결정의 근거를 기록해서 6개월 뒤에도 '왜?'에 답할 수 있다는 것이다."

— Software Engineering at Google (O'Reilly, 2020)

AI 에이전트 시대의 DESIGN.md: Claude Code, Cursor, Copilot 같은 AI 에이전트는 코드를 읽어서 "무엇"은 파악할 수 있다. 하지만 "왜 이 구조인가", "무엇을 하면 안 되는가"는 코드에서 유추하기 어렵다. DESIGN.md가 이 간극을 메운다.

프로젝트 문서의 역할 분담

각 마크다운 파일이 담당하는 영역이 다르다. 겹치지 않게 분리하면 효과적이다.

DESIGN.md 핵심 섹션

Google의 디자인 문서 전통을 기반으로, AI 에이전트 시대에 맞게 구성한 섹션 가이드

1
Overview
시스템이 무엇이고 왜 존재하는지 한 문단으로 요약. 처음 읽는 사람이 10초 안에 전체 그림을 잡을 수 있어야 한다.
2
Goals & Non-Goals
이 시스템이 달성하려는 것과 명시적으로 하지 않으려는 것. Non-Goals가 특히 중요하다 — AI 에이전트가 범위를 벗어나는 변경을 제안하는 걸 막아준다.
3
Architecture
주요 컴포넌트와 그 관계. ASCII 다이어그램 권장 — AI 에이전트가 이미지를 읽을 수 없으므로 텍스트 기반이 유리하다.
4
Design Decisions
핵심 설계 결정, 고려한 대안, 선택 이유. 각 결정마다 Context → Decision → Alternatives → Rationale 구조로 기록한다.
5
Data Flow
데이터가 시스템을 통과하는 경로. 입력 → 처리 → 저장 → 출력의 흐름을 단계별로 서술한다.
6
Cross-cutting Concerns
보안, 성능, 확장성, 에러 처리 등 여러 컴포넌트에 걸친 관심사. 기술 부채로 남기기 쉬운 영역이므로 명시적으로 기록한다.
7
Constraints
기술적·사업적 제약 조건. "PostgreSQL만 사용", "브라우저 호환성은 최신 2버전만", "번들 크기 200KB 이하" 같은 명시적 제약.
8
Future Considerations
알려진 한계와 향후 계획. "현재는 단일 리전이지만 멀티 리전 확장 예정" 같은 로드맵 맥락.

실전 예제

SaaS 플랫폼 DESIGN.md

# DESIGN.md — TaskFlow

## Overview
TaskFlow는 팀 프로젝트 관리 SaaS다. Jira의 복잡함과 Trello의 단순함
사이에서 중간 지점을 목표로 한다. Next.js App Router +
Prisma + PostgreSQL 기반.

## Goals
- 5명 이하 소규모 팀을 위한 태스크 관리
- 실시간 협업 (WebSocket 기반)
- 모바일 웹에서도 완전한 기능 사용 가능

## Non-Goals
- 엔터프라이즈 기능 (SAML, SCIM, 감사 로그)
- 데스크톱/모바일 네이티브 앱
- Gantt 차트, 리소스 관리 같은 PM 고급 기능
- 100명 이상의 팀 지원

## Architecture

┌─────────────┐     ┌──────────────┐     ┌──────────┐
│   Next.js   │────▶│   API Routes │────▶│ PostgreSQL│
│  (App Router)│     │  (tRPC)      │     │ (Prisma) │
└──────┬──────┘     └──────┬───────┘     └──────────┘
       │                   │
       │            ┌──────▼───────┐
       └───────────▶│  WebSocket   │
                    │  (Pusher)    │
                    └──────────────┘

### 컴포넌트 책임
- Next.js App Router: UI 렌더링, 라우팅, SSR
- tRPC: 타입 안전한 API 레이어 (REST 대신 선택)
- Prisma: ORM, 마이그레이션, 타입 생성
- Pusher: 실시간 이벤트 (자체 WS 서버 대신 — 운영 부담 회피)

## Design Decisions

### tRPC vs REST vs GraphQL
- Context: 프론트/백엔드 모두 TypeScript, 5명 팀
- Decision: tRPC 사용
- Alternatives: REST (범용적), GraphQL (유연한 쿼리)
- Rationale: 팀 규모가 작고 모노레포라서 E2E 타입 안전성이
  가장 큰 이점. GraphQL은 이 규모에서 오버엔지니어링.

### Pusher vs 자체 WebSocket 서버
- Context: 실시간 기능 필요, 인프라 인력 없음
- Decision: Pusher 사용
- Rationale: 자체 WS 서버는 운영 부담 큼. Pusher의
  무료 티어로 초기 사용자 충분히 처리 가능.

## Constraints
- Vercel 배포: Edge Runtime 제약 (Node.js API 일부 사용 불가)
- 번들 크기: First Load JS 200KB 이하 유지
- DB: Neon PostgreSQL (Serverless, connection pooling 필수)
- 인증: NextAuth.js (OAuth만, 자체 패스워드 인증 안 함)

## Future Considerations
- Pusher → 자체 WebSocket 전환 검토 (MAU 10K 초과 시)
- i18n 지원 (한/영/일) — 현재는 한국어만
- 팀 간 프로젝트 공유 기능 — 현재 아키텍처에서 가능

CLAUDE.md에서 DESIGN.md 연동

# CLAUDE.md

## 아키텍처
아키텍처 결정과 그 이유는 DESIGN.md를 참조하라.
@DESIGN.md

## 중요
- DESIGN.md의 Non-Goals에 명시된 기능을 제안하지 않는다
- 아키텍처 변경이 필요한 경우 DESIGN.md를 먼저 확인한다
- 새로운 설계 결정은 DESIGN.md의 Design Decisions 섹션에 추가한다

AI 코딩 도구와의 연동

DESIGN.md를 각 AI 도구에서 활용하는 방법

Claude Code

CLAUDE.md에서 @DESIGN.md로 import. 아키텍처 변경 전 자동 참조. Skills에서 디자인 리뷰 워크플로우 정의 가능.

Cursor

.cursor/rules/에서 DESIGN.md를 참조하도록 지시. @docs 기능으로 인덱싱하면 자동 검색 대상이 된다.

Aider

--read DESIGN.md 플래그로 읽기 전용 컨텍스트에 추가. 아키텍처 인식 상태에서 코드 수정.

Copilot

.github/copilot-instructions.md에서 DESIGN.md를 참조하도록 명시. 조직 레벨 지시에 아키텍처 원칙 포함.

주의: 오래된 DESIGN.md는 없는 것보다 나쁘다. AI 에이전트가 현재와 다른 아키텍처를 기준으로 코드를 수정할 수 있다. 아키텍처가 바뀌면 DESIGN.md도 반드시 갱신한다.

ADR: Architecture Decision Records

DESIGN.md의 보완재. 개별 설계 결정을 독립 문서로 기록하는 방식.

DESIGN.md + ADR 조합

DESIGN.md가 "전체 아키텍처의 현재 상태"라면, ADR은 "개별 결정의 히스토리"다. DESIGN.md는 최신 상태만 유지하고, 결정의 배경과 대안은 ADR에 기록하면 깔끔하다.

  • DESIGN.md — 시스템의 현재 아키텍처 스냅샷 (최신 상태)
  • docs/adr/ — 개별 결정의 전체 맥락과 히스토리
  • DESIGN.md에서 각 결정에 대해 ADR 링크로 연결
  • ADR은 한 번 쓰면 수정하지 않는다 (새 결정은 새 ADR)
# 디렉토리 구조
my-project/
├── DESIGN.md                # 전체 아키텍처 (현재 상태)
├── docs/adr/                # 개별 결정 히스토리
│   ├── 001-use-trpc.md
│   ├── 002-pusher-over-ws.md
│   ├── 003-neon-postgresql.md
│   └── template.md          # ADR 템플릿
├── CLAUDE.md
└── ...

ADR 템플릿

# docs/adr/template.md

# ADR-{번호}: {제목}

## Status
Accepted | Proposed | Deprecated | Superseded by ADR-{번호}

## Context
이 결정이 필요한 배경과 상황.

## Decision
무엇을 결정했는가.

## Alternatives Considered
검토한 대안과 각각의 장단점.

## Consequences
이 결정으로 인한 긍정적/부정적 영향.

## Date
2026-04-13

작성 가이드

DO
  • ✓ "왜"에 집중 — 코드가 말해주지 않는 것
  • ✓ Non-Goals를 명시적으로 기록
  • ✓ ASCII 다이어그램 사용 (AI 판독 가능)
  • ✓ 제약 조건을 구체적으로 (수치 포함)
  • ✓ 아키텍처 변경 시 DESIGN.md도 갱신
  • ✓ 500~2000줄 이내로 유지
DON'T
  • ✗ 코드를 읽으면 알 수 있는 내용 반복
  • ✗ 이미지만으로 다이어그램 표현
  • ✗ API 레퍼런스 문서를 대체하려 시도
  • ✗ 작성 후 방치 (stale한 문서는 유해)
  • ✗ 구현 세부사항 (코드 레벨) 기록
  • ✗ CLAUDE.md와 내용 중복
유사한 파일명: 일부 프로젝트에서는 ARCHITECTURE.md(rust-analyzer에서 유래)나 CONTRIBUTING.md의 아키텍처 섹션으로 같은 역할을 수행한다. 파일명보다 내용이 중요하다 — 핵심은 "왜 이 구조인가"를 기록하는 것.
