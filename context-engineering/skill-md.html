Skills 완전 가이드

반복되는 워크플로우를 스킬 파일로 정의하고,
슬래시 명령 한 번으로 호출한다.

Skills란?

Skills는 Claude Code의 .claude/skills/ 디렉토리에 저장하는 마크다운 파일이다. 각 파일이 하나의 슬래시 명령이 되어, 특정 작업을 수행할 때 필요한 지시를 온디맨드로 주입한다. CLAUDE.md가 "항상 적용되는 규칙"이라면, Skills는 "필요할 때 꺼내 쓰는 도구"다.

CLAUDE.md

  • 매 대화 시작 시 항상 로드
  • 프로젝트 전반의 규칙과 컨벤션
  • 빌드 명령어, 코드 스타일, 아키텍처
  • 200줄 이하 권장 (항상 컨텍스트 차지)
  • "이 프로젝트에서는 이렇게 해라"

Skills

  • /명령어로 호출할 때만 로드
  • 특정 작업을 위한 절차와 워크플로우
  • 배포, 마이그레이션, 코드 리뷰, 페이지 생성
  • 길이 제한 없음 (필요할 때만 주입)
  • "이 작업은 이 순서로 해라"
핵심 차이: CLAUDE.md에 모든 걸 넣으면 컨텍스트 윈도우가 낭비된다. 배포 절차를 코드 작성할 때마다 로드할 필요가 없다. 자주 쓰지 않지만 정확한 절차가 필요한 작업은 Skills로 분리하면 된다.

동작 원리

스킬 파일의 생명 주기

1. 파일 생성
.claude/skills/deploy.md
마크다운 파일을 .claude/skills/ 아래에 저장한다. 파일명이 곧 명령어 이름이 된다.
2. 명령어 등록
/deploy
Claude Code가 디렉토리를 스캔하고 자동으로 슬래시 명령으로 등록한다. 별도 설정 불필요.
3. 호출
사용자가 /deploy 입력
스킬 파일의 내용이 대화 컨텍스트에 주입된다. Claude가 지시를 따라 작업을 수행한다.
4. 실행
단계별 수행
스킬에 정의된 절차를 순서대로 실행한다. 필요하면 사용자에게 확인을 요청한다.

디렉토리 구조

# 프로젝트 루트
my-project/
├── CLAUDE.md                  # 항상 적용되는 규칙
├── .claude/
│   └── skills/                # 스킬 디렉토리
│       ├── deploy.md          # /deploy 명령
│       ├── review.md          # /review 명령
│       ├── create-page.md     # /create-page 명령
│       ├── db-migrate.md      # /db-migrate 명령
│       └── add-feature.md     # /add-feature 명령
└── src/

# 글로벌 스킬 (모든 프로젝트에서 사용)
~/.claude/
└── skills/
    ├── commit.md              # /commit — 어디서든 사용
    └── pr.md                  # /pr — 어디서든 사용

스킬 파일 작성법

스킬 파일은 자유 형식 마크다운이다. 정해진 스키마 없이, Claude에게 보내는 지시를 그대로 작성하면 된다.

기본 구조

# .claude/skills/deploy.md

## 목적
프로덕션 환경에 배포하는 절차. 빌드 검증부터 배포 완료 확인까지.

## 사전 조건
- main 브랜치에서 작업 중이어야 한다
- 커밋되지 않은 변경이 없어야 한다

## 절차
1. git status로 클린 상태 확인
2. pnpm build로 빌드 성공 확인
3. pnpm test로 전체 테스트 통과 확인
4. 빌드/테스트 모두 통과하면 git tag v{버전} 생성
5. git push origin main --tags 실행
6. GitHub Actions 배포 완료 대기
7. 배포 결과 요약 출력

## 주의사항
- 테스트 실패 시 배포를 중단하고 실패 원인을 보고한다
- 태그 버전은 package.json의 version 필드를 따른다
- 배포 전 반드시 사용자 확인을 받는다
작성 팁: "똑똑한 동료에게 작업을 인수인계하듯" 작성한다. 배경 지식을 전부 설명할 필요 없고, 핵심 절차와 주의사항에 집중한다. Claude는 코드를 읽을 수 있으므로 코드에서 알 수 있는 내용은 생략해도 된다.

코드 리뷰 스킬

# .claude/skills/review.md

현재 브랜치의 변경사항을 리뷰한다.

## 리뷰 절차
1. git diff main...HEAD로 변경 범위 확인
2. 변경된 파일을 하나씩 읽고 다음 관점에서 검토:
   - 버그 가능성 (null 접근, 경계값, 동시성)
   - 성능 문제 (N+1 쿼리, 불필요한 렌더링)
   - 보안 취약점 (인젝션, XSS, 인증 누락)
   - 코드 가독성과 네이밍
3. 발견한 이슈를 심각도별로 분류:
   - CRITICAL: 반드시 수정 (버그, 보안)
   - SUGGESTION: 수정 권장 (가독성, 성능)
   - NITPICK: 사소한 개선 (스타일, 네이밍)
4. 각 이슈에 파일명:라인번호와 수정 제안을 포함

## 출력 형식
이슈를 심각도별로 그룹핑해서 출력한다.
이슈가 없으면 "LGTM" 으로 마무리한다.

페이지 생성 스킬 (정적 사이트용)

# .claude/skills/create-page.md

새로운 HTML 페이지를 프로젝트에 추가한다.

## 절차
1. 사용자에게 페이지 제목과 설명을 확인
2. 기존 페이지(예: about.html)를 템플릿으로 참조
3. 동일한 nav, footer, CSS 변수 체계를 유지
4. 새 페이지의 nav에서 해당 페이지를 active로 표시
5. IMPORTANT: 기존 모든 페이지의 nav에 새 페이지 링크를 추가
6. footer에 트위터 공유 버튼 포함 (페이지 URL + 제목)

## 디자인 규칙
- Hero 배경: 다크 그라데이션 (기존 페이지와 톤 통일)
- 폰트: Noto Sans KR + JetBrains Mono
- 반응형: 768px 브레이크포인트
- 색상 체계: CSS Custom Properties 사용

DB 마이그레이션 스킬

# .claude/skills/db-migrate.md

데이터베이스 스키마 변경을 안전하게 수행한다.

## 절차
1. 변경할 스키마 내용을 사용자에게 확인
2. prisma/schema.prisma 수정
3. pnpm prisma migrate dev --name {설명} 실행
4. 마이그레이션 SQL 파일 내용 검토
5. pnpm prisma generate로 클라이언트 재생성
6. 영향받는 코드를 찾아서 타입 업데이트

## 안전 규칙
- 컬럼 삭제 시 반드시 사용자 확인
- NOT NULL 컬럼 추가 시 기본값 필수
- 인덱스 추가/삭제는 마이그레이션 분리
- 프로덕션 마이그레이션은 이 스킬 범위 밖 (수동 처리)

커밋 스킬

# .claude/skills/commit.md

현재 변경사항을 커밋한다.

## 절차
1. git status와 git diff로 변경 내용 파악
2. 변경 내용을 분석해서 커밋 메시지 초안 작성
3. Conventional Commits 형식 사용:
   - feat: 새 기능
   - fix: 버그 수정
   - refactor: 리팩토링
   - docs: 문서
   - chore: 기타
4. 제목은 50자 이내, 본문은 한국어
5. 민감한 파일(.env, credentials)이 포함되어 있으면 경고
6. 사용자 확인 후 커밋 실행

활용 패턴

스킬을 효과적으로 활용하는 패턴들

/deploy

배포 절차 자동화. 빌드 검증, 테스트 실행, 태그 생성, 푸시까지 일관된 순서로 수행.

/review

코드 리뷰 체크리스트 기반 검토. 심각도 분류, 수정 제안, 파일:라인 참조.

/create-page

새 페이지 스캐폴딩. 템플릿 복제, nav 일괄 업데이트, 공유 버튼 자동 생성.

/db-migrate

DB 마이그레이션 안전 수행. 스키마 수정, 마이그레이션 생성, 영향 코드 업데이트.

/add-api

API 엔드포인트 추가. 라우트, 핸들러, 타입, 테스트까지 일괄 생성.

/onboard

프로젝트 온보딩. 구조 설명, 핵심 파일 안내, 개발 환경 체크.

베스트 프랙티스

1
한 스킬 = 한 작업
하나의 스킬 파일에 여러 작업을 넣지 않는다. "배포 + 롤백"이 아니라 /deploy와 /rollback으로 분리한다.
2
파일명 = 동사 또는 동사구
deploy.md, create-page.md, review-pr.md 처럼 작업을 설명하는 동사로 이름 짓는다. 사용자가 /로 호출할 때 직관적이다.
3
절차를 구체적으로
실행할 명령어, 확인할 조건, 실패 시 대응을 구체적으로 적는다. 추상적인 지시("테스트를 돌려라")보다 구체적인 명령어(pnpm vitest run)가 낫다.
4
CLAUDE.md 규칙을 반복하지 않기
스킬 파일에서 CLAUDE.md에 이미 정의된 컨벤션을 반복할 필요 없다. Claude는 CLAUDE.md를 이미 읽고 있으므로 스킬에서는 작업 절차에만 집중한다.
5
글로벌 vs 프로젝트 분리
프로젝트에 무관한 범용 스킬(커밋, PR 생성 등)은 ~/.claude/skills/에 두고, 프로젝트 고유 스킬(배포, 마이그레이션 등)은 프로젝트의 .claude/skills/에 둔다.
흔한 실수: CLAUDE.md에 모든 걸 넣는 것. 200줄 제한 권장은 "항상 로드되는 것"의 크기를 제한하기 위함이다. 배포 절차 30줄, DB 마이그레이션 20줄, 코드 리뷰 체크리스트 25줄 — 이런 건 Skills로 분리해야 CLAUDE.md가 가볍게 유지된다.
